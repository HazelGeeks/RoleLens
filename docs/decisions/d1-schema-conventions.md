# D1 Schema Conventions

RoleLens uses Cloudflare D1 as the canonical relational store. This document defines the naming and shape rules for current and future tables.

## Current Table Inventory

User-owned product data:

- `persistent_jobs`
- `persistent_job_notes`
- `persistent_job_create_requests`
- `persistent_goals`
- `persistent_goal_followups`

Authentication:

- `auth_users`
- `auth_sessions`

Feed snapshots:

- `feed_import_snapshots`

Obsolete tables:

- `scraped_feed_snapshots`
- `scraped_feed_snapshot_chunks`

These obsolete tables are removed by `migrations/0005_drop_legacy_scraped_feed_snapshots.sql`.

## Naming Rules

- Use lowercase `snake_case` for all tables, columns, and indexes.
- Use plural table names for entity collections: `auth_users`, `persistent_jobs`, `persistent_goals`.
- Use domain prefixes:
  - `auth_` for authentication/account/session records.
  - `persistent_` for user-owned application data exposed through the persistence API.
  - `feed_` for imported feed snapshots and sync metadata.
- Do not introduce new `scraped_` tables. Scraping or ingestion output must land in D1 through `feed_import_snapshots` unless a future migration introduces a clearly named `feed_*` table.
- Name indexes as `idx_{table}_{columns_or_purpose}`.
- Name foreign key columns as `{parent_singular}_id`, for example `job_id`, `goal_id`, `user_id`.

## Column Shape Rules

- Every user-owned table must include `user_id` unless it is a global lookup or system table.
- Primary keys are `TEXT` ids generated by the application.
- Timestamps are ISO strings stored as `TEXT` and named with `_at`, for example `created_at`, `updated_at`, `expires_at`.
- Date-only values are stored as `TEXT` in `YYYY-MM-DD` format and named with `_date`, for example `follow_up_date`.
- JSON payload columns must end in `_json`, for example `tags_json` and `snapshot_json`.
- Versioned mutable records use an integer `version` column and update with an optimistic concurrency check.

## Relationship Rules

- Child tables that are scoped to a parent and a user should store both the parent id and `user_id`.
- Cascading deletes are allowed when the child cannot stand alone, for example:
  - `auth_sessions.user_id -> auth_users.id`
  - `persistent_goal_followups.goal_id -> persistent_goals.id`
- Queries must always include `user_id` for user-owned data access.

## Migration Rules

- Never edit an applied migration. Add a new numbered migration.
- Keep `database/schema.sql` as the canonical snapshot of the final expected schema.
- Cleanup migrations should use `DROP ... IF EXISTS` when removing obsolete tables or indexes.
- Apply migrations locally before handoff with `npm run db:schema:local`.
- Apply migrations to production before deploy with `npm run db:schema:prod`.
